# タブレノ ターゲットユーザー・機能整理

Last updated: 2026-07-05

## 目的

タブレノのターゲットユーザーを定め、そのユーザーが使う機能、見える情報、できないことを整理する。
この文書は、セッション管理を中心にした初期導入設計、権限設計、実装優先順位の判断材料として使う。

## プロダクトの位置づけ

タブレノは、CoC卓の予定・参加者・キャラ・シナリオ情報をまとめて共有できる卓管理ツールである。

以下のようには打ち出さない。

- 汎用TRPG管理ツール
- 日程調整専用ツール
- キャラシ専用ツール
- Discord代替ツール

タブレノは Discord やキャラシサイトを置き換えるものではなく、CoC卓の情報をまとめ、共有しやすくする中心置き場として位置づける。

## ターゲット構造

### 主ターゲット

クトゥルフ神話TRPGのKP/GM。特に固定卓や複数卓を継続運営している人。

最重要ペルソナは、複数のCoC卓をDiscordで運営しているKP/GMである。
固定メンバーと単発参加者が混在し、予定・参加者・キャラ・シナリオ・HO情報を毎回整理している。

### 入口ターゲット

タブレノの全機能を最初から使うユーザーだけでなく、一部機能だけを使うユーザーも入口として想定する。

- セッション管理から入るユーザー
- キャラシ管理から入るユーザー
- シナリオ / HO管理から入るユーザー
- グループ管理から入るユーザー

ただし主導線はセッション管理に寄せる。他の入口から入っても、最終的にセッションへ接続できるようにする。

## 最重要課題

セッションごとの情報が Discord、キャラシサイト、メモ、カレンダーに分散していて、開催前に毎回確認・再共有が発生すること。

タブレノの初期価値は、卓情報の一元化と再共有の手間削減である。

## 初回導入方針

### 第一入口

第一入口はセッション管理・日程管理とする。

ただし初期価値は「日程を決めること」ではなく、決まったセッション予定を管理・共有することに置く。

### 主導線

主導線は以下とする。

1. グループを作る
2. セッションを作る
3. 参加者を追加する
4. 共有する

単発卓や個人メモ用途のために、「セッションだけ作る」ショートカットも用意する。

### 最小入力

グループ作成の必須入力はグループ名だけにする。
説明、公開範囲、メンバー招待は後から設定できる。

セッション作成の必須入力はタイトルと日時だけにする。
シナリオ、参加者、場所 / URL、概要、HOは後から追加できる。

セッション作成直後は、参加者の追加と共有リンクの発行を促す。

## セッションの扱い

セッションは「自分が参加する予定」だけでなく、「グループで管理する卓情報」としても扱う。

- KP/GM以外でもセッションを作成できる
- PLがセッションを作り、後からKP/GMを誘う導線を残す
- 作成者自身が参加者でなくても、グループのセッション記録として登録できる
- セッション作成者またはKP/GMは、グループ内ユーザーを参加者・KP/GMとして紐づけられる
- 参加者は登録ユーザーでも、名前だけの仮参加者でもよい

## 共有リンク方針

基本情報は共有リンクでログインなし閲覧可とする。
参加者管理、キャラ紐づけ、秘匿HO、編集はログイン必須とする。

ログインなし共有リンクで見せる情報は以下に限定する。

- 日時
- タイトル
- シナリオ公開概要
- 開催場所 / 参加URL
- 公開参加者名
- 連絡事項

共有リンクを開いたPLは、まず閲覧のみできる。
参加者として自分のキャラを紐づける、秘匿HOを見る、自分の予定表に保存する場合はログイン / 登録へ誘導する。

秘匿HO、GMメモ、非公開参加者情報、内部ユーザーID、メールアドレスは共有リンクでは出さない。

## ユーザー分類

| ユーザー分類 | 定義 | 主な価値 |
| --- | --- | --- |
| KP/GM | 卓を運営する主ユーザー | 卓情報、参加者、シナリオ、HOを一元管理できる |
| PL | セッションに参加するユーザー | 自分の予定、キャラ、HO、参加履歴を確認できる |
| セッション作成者 | KP/GMではないがセッション情報を作る人 | PL起点や記録目的で卓情報を作成・共有できる |
| グループ管理者 | 固定卓 / サークル単位で管理する人 | グループ、メンバー、所属セッションを管理補助できる |
| 未ログイン閲覧者 | 共有リンクから基本情報だけ見る人 | アカウント登録前に卓情報を確認できる |

## 初期対応と将来対応

### 初期対応

初期導入では以下に絞る。

- セッション作成
- 参加者管理
- 共有リンク
- 基本情報閲覧
- キャラ紐づけ
- HO管理

### 将来対応

以下は便利ではあるが、初期導入の主価値からは外す。

- 統計
- 外部連携
- 高度な通知
- 日程調整

実装済みの機能であっても、初期訴求や初回導線では前面に出さない。

## 要調整項目

## 権限モデル刷新方針

権限周りは既存互換を優先せず、メンテナンス性と安全性を優先して一新する。
リリースは、要件中の権限周りを実装し切るまで控える。

### セッション本体

`TRPGSession` はセッションそのものを表す。

- タイトル、日時、グループ、公開範囲、作成者などを持つ。
- GM未定セッションを許可する。
- 既存の `gm` は必須にしない。必要なら表示用の主KP/GMとして扱うが、権限判定の主軸にはしない。
- PLが先にセッションを作って後からKP/GMを誘う導線を正式に扱う。

### 参加者と管理権限の分離

`SessionParticipant` は卓への参加と卓内役割を表す。
`SessionPermission` は編集、共有、秘匿管理などの管理権限を表す。

参加者であることと、管理権限を持つことは分離する。
作成者が参加しないセッション、PL起点セッション、グループ記録用セッション、グループ管理者の管理補助を自然に扱うためである。

### SessionPermission

`SessionPermission` はロール制 + 必要最小限のフラグで実装する。
1ユーザーに複数ロールを付与できる。

初期ロールは以下の4つとする。

| ロール | 意味 | 秘匿情報 |
| --- | --- | --- |
| `owner` | セッション作成者 / 管理責任者。基本情報、参加者、共有リンク、権限管理を扱う | 見えない。秘匿情報には別途 `secret_keeper` が必要 |
| `manager` | 基本情報、参加者、共有リンクを管理する | 見えない |
| `secret_keeper` | シナリオ本文、秘匿HO、GMメモを管理する | 見える |
| `viewer` | 管理補助用の閲覧のみ | 原則見えない |

セッション作成者には `owner` を自動付与する。
作成時に自分をKP/GMとして設定した場合のみ、あわせて `secret_keeper` も自動付与する。

`owner` は基本情報と権限管理の責任者であり、秘匿情報を見るロールではない。
PLがセッションを作成してKP/GMを誘うケースで、作成者が秘匿HOやGMメモを見えてしまう事故を防ぐ。

### KP/GMと secret_keeper

`SessionParticipantRole(role="gm")` になったユーザーには、原則 `secret_keeper` を自動付与する。
ただし、共同GM、見学GM、管理補助などの例外に対応するため、削除 / 降格できるようにする。

### グループ管理者

グループ管理者は、明示的な `SessionPermission` がなくても、グループ所属セッションに対して以下を管理できる。

- セッション基本情報
- 参加者
- 共有リンク

ただし、グループ管理者であるだけでは秘匿情報は見えない。
秘匿情報を見るには、個別セッションで `secret_keeper` を付与する。

### 権限判定サービス

既存の大きな `can_manage_session` は廃止する。
内部互換関数としても残さず、新規のユースケース単位の関数へ置き換える。

候補:

- `can_view_session_basic(user, session)`
- `can_edit_session_basic(user, session)`
- `can_manage_participants(user, session)`
- `can_manage_share_links(user, session)`
- `can_manage_permissions(user, session)`
- `can_view_secret_content(user, session)`
- `can_manage_secret_content(user, session)`

基本情報管理、参加者管理、共有リンク管理、秘匿情報閲覧を混同しないことを最優先する。

### 参加者ロール

参加者ロールも複数許可する。
`SessionParticipant.role` のような単一フィールドには寄せず、`SessionParticipantRole` として分離する。

初期ロールは以下の3つとする。

- `gm`
- `player`
- `observer`

同じユーザーが `gm` と `player` を兼ねられる。
作成者や管理者は参加者ロールではなく、`SessionPermission` で表現する。

仮参加者にも `gm`, `player`, `observer` を付与できる。
後から本人アカウントに紐づけてもロールを維持する。

### ParticipantIdentity

`ParticipantIdentity` をグループ内人物マスタとして拡張する。

- 仮メンバー / 仮参加者は `ParticipantIdentity` に紐づける。
- 同じ人を複数セッションで再利用できる。
- 後からタブレノユーザーに紐づけても、過去セッションとのつながりを保つ。
- `ParticipantIdentity.group` は nullable にする。
- グループセッションでは `group` を持つ。
- 個人セッションでは `created_by` または session スコープで扱う。
- 実ユーザーとの現在紐づけは1つにする。
- 紐づけ変更履歴は監査ログに残す。

### 仮参加者 / 仮メンバーの引き取り

仮参加者 / 仮メンバーの引き取りは原則承認必須にする。

基本フロー:

1. 本人が招待リンクからログイン / 登録する。
2. 「自分です」と申請する。
3. グループ管理者、セッション作成者、KP/GMが承認する。
4. 承認後に仮参加者 / 仮メンバーが本ユーザーへ紐づく。

ただし、招待リンクに特定の `ParticipantIdentity` が紐づいている場合は、発行者が許可した安全な直接claimとして扱える余地を残す。

直接claimを許可するリンクの必須条件:

- `identity_id` が指定されている
- 有効期限がある
- 1回限り
- 発行者が `allow_direct_claim=true` を明示している

それ以外は承認申請に回す。

### セッション作成者とKP/GMの分離

現状の `TRPGSession` は `gm` と `created_by` を持つが、API作成時は作成者がGMとして保存される。

必要な状態:

- 作成者はセッション基本情報、参加者、共有リンクを管理できる
- KP/GMはそれに加えてシナリオ、HO、GMメモ、PLキャラ確認を管理できる
- 作成者はGM、PL、GM+PL、作成者のみの状態を取れる
- 作成者権限だけでは、KP/GM専用情報を閲覧 / 編集できない

### グループ仮メンバー

必要な状態:

- グループに名前だけの仮メンバーを追加できる
- 仮メンバーをセッション参加者として流用できる
- 後から本人のタブレノアカウントに紐づけられる

### 仮参加者の承認式引き取り

必要な状態:

- 本人が招待リンクからログイン / 登録して「自分です」と申請できる
- グループ管理者、セッション作成者、KP/GMが承認できる
- 承認後に仮参加者 / 仮メンバーが本ユーザーへ紐づく
- 既存登録ユーザーは、作成者またはKP/GMがグループ内ユーザー一覧から直接紐づけられる

## セッション詳細ページの優先表示

セッション詳細ページでは以下を優先して表示する。

- 日時
- タイトル
- 参加者
- 開催場所 / URL
- シナリオ概要
- キャラ紐づけ状況
- HO状況
- 連絡事項

このページが、KP/GMの開催前確認とPLへの共有の中心になる。
